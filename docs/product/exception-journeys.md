## Exception Principles

- Exceptions are first-class workflows.
- No exception is resolved by silent data editing.
- Every exception has an owner.
- Every exception has a patient-safe fallback.
- Every exception has an audit trail.
- Every exception has a closure condition.
- Clinical and privacy incidents may require separate governance review.
- Emergency safety action precedes commercial resolution.
- Routine operational correction must use workflow, policy, and audit mechanisms rather than direct production database editing.

## Exception Schema

Every exception entry below includes: exception ID, name, related journey IDs, trigger, detection source, scope, provisional severity/priority, immediate containment, patient or participant communication, human owner, escalation owner, data access required, data access prohibited, compensating action, recovery steps, financial effect, clinical-safety effect, audit events, closure condition, reopening condition, later test cases, and related open questions.

## Provider-Disclosure Incident Flow

This dedicated flow applies to EXC-047 and any suspected pharmacy or laboratory detail leakage before the approved successful-payment event.

1. Immediate containment: disable or gate the affected endpoint, projection, UI route, fixture, analytics stream, or feature flag.
2. Evidence preservation: save the minimum necessary response, trace, log, screenshot, or browser artifact in controlled incident storage.
3. Client artifact containment: remove protected details from caches, hydration payloads, browser storage, test snapshots, screenshots, and traces where possible without destroying evidence.
4. Security and privacy escalation: notify Security lead and Privacy counsel; assign incident owner and product owner.
5. Patient-impact assessment: determine affected actors, orders, tenants, and whether any provider identity/location could be derived.
6. Operational pause: suspend affected pharmacy/lab matching path if leakage cannot be isolated.
7. Corrective action: update server-side projection and policy checks. Visual hiding is not a valid fix.
8. Corrective testing: run future API-contract, security/privacy, Playwright, and interactive browser checks proving prohibited fields are absent from HTML, DOM, hidden DOM, accessibility tree, network, hydration, storage, cache, analytics, error reporting, logs, screenshots, browser traces, source output, and client-visible fixtures.
9. Communication decision: Privacy counsel and legal counsel decide whether and how to notify affected users or partners.
10. Closure: close only after evidence is preserved, impact is assessed, corrective tests pass, and owner approvals are recorded.
11. Reopening: reopen on any recurrence, any newly discovered channel, any direct URL/IDOR path, or any analytics/log/cache exposure.
