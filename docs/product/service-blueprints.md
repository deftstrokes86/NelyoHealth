## Blueprint Method

Each blueprint row uses these lanes:

1. Customer or participant action
2. Frontstage user interface
3. API or application decision
4. Domain or workflow decision
5. Human operations
6. Provider or external partner
7. Data created
8. Data disclosed
9. Authorization or consent
10. Payment or coverage
11. Notification
12. Audit
13. Failure detection
14. Recovery action
15. Completion or reopening

Blueprints are conceptual P00-05 design artifacts. Final states, transition names, timers, retries, idempotency rules, and workflow diagrams belong to P00-07.

## Frontstage and Backstage Boundary

- A matching engine may use pharmacy or laboratory location internally.
- The pre-payment patient interface may not receive or infer pharmacy/laboratory address, coordinates, distance, branch, map pin, directions, contact details, photographs, links, pickup/collection instructions, internal identifiers, or derivable metadata.
- Pre-payment API and browser payloads must be sanitized at the server boundary. The frontend must not receive a full provider object and hide fields.
- Pre-payment discovery and post-payment fulfilment detail retrieval are separate conceptual operations.
- Post-payment detail retrieval must validate actor, patient, tenant, selected order, selected provider, and the successful-payment event later approved in P00-13.
- Emergency facility guidance, routine hospital referral details, and clinically required transfer information are not blocked by the pharmacy/laboratory obscuration rule.
