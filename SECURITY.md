# Security Policy

## Supported scope

This repository supports foundation-stage documentation, tooling, synthetic browser checks, design foundation packages, and local verification scripts. It does not contain production product code, production credentials, or production data handling.

## Reporting a vulnerability

Do not open public issues for vulnerabilities. Prefer GitHub private vulnerability reporting or a private security advisory when enabled. If private vulnerability reporting is not verified, use an established private repository-owner channel designated by the external orchestrator. Do not invent or use undocumented contact information.

## Useful report information

Include sanitized information only:

- Affected file, workflow, package, or tool.
- Minimal reproduction using synthetic data.
- Expected and actual behavior.
- Security boundary affected.
- Potential impact.
- Suggested fix, if known.

Do not include real patient, provider, clinical, payment, credential, organization, employer, HMO, sponsor, guardian, or production data.

## Coordinated disclosure

Response targets are not contractual. Security reports may trigger incident handling, privacy review, dependency review, or domain-owner escalation. Do not publicly disclose exploit details until the repository owner and relevant domain owners have reviewed the issue.

## Emergency care

Emergency patient-care matters do not belong in the repository security channel. This repository is not an emergency-care service.

## Governance references

Security handling is subordinate to NelyoHealth locked requirements, privacy governance, provider-disclosure controls, dependency governance, ADRs, and external domain approvals. Codex output is not security approval.
